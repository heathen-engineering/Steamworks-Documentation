Detailed Description

This replaces the SteamManager concept from classic Steamworks.NET

The SteamworksFoundationManager initalizes the client SteamAPI and handles callbacks for the system. For the convenance of users using a singleton model this class also provides a Instance static member and wraps all major funcitons and event of the SteamSettings object.

The SteamworksFoundationManager is the core compoenent to the Heathen Steamworks kit and replaces the funcitonality present in Steamworks.NET's SteamManager. The primary funciton of the manager is to operate the update loop required by the Steam API and to handle and direct callbacks from the Steam API.

It is strongly advised that you never unload or reload the SteamworksFoundationManager for example you should not place the SteamworksFoundationManager in your title scene because that scene will be unloaded and reloaded multiple times. Even if you mark the object as Do Not Destroy on Load, on reload of the title or similar scene Unity will create a second SteamworksFoundationManager creating issues with the memory it manages.

The recomended approch is place your SteamworksFoundationManager and any other "manager" in a bootstrap scene that loads first and is never reloaded through the life of the game. This will help insure that through the life of your users play session 1 and exsactly 1 SteamworksFoundationManager is created and never destroyed. While there are other approches you could take to insure this using a simple bootstrap scene is typically the simplest. For more information on this subject see . This article referes to another tool available from Heathen Engineering however the concepts within apply rather or not your using that tool.

◆ Instance

SteamworksFoundationManager HeathenEngineering.SteamApi.Foundation.SteamworksFoundationManager.Instance
staticget

For use with the singleton approch

Heathen Engineering recomends the use of direct references. Note that all required funcitonality of the SteamworksFoundationManager is available in the SteamSettings scriptable object which can be referenced on any game object directly as it is not a scene object it is not limited to references within the current scene.

The singleton approch is viable however not recomended as this creates a dependency between GameObejcts and consaquently execution timing that is unnessiary. The recomended solution is to use a reference to the SteamSettings object directly ... e.g. for any script that needs to access Steam features simply give it a reference of

public SteamSettings settings;

You can now reference the settings scriptable object and use it directly in that script. As the Scriptable Object is defined in the asset database it is always available regardless of execution order and does not requrie GameObject scene references.
